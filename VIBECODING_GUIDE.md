# NEKO_ERP Vibecoding 开发总指导

- 版本: v2.0
- 时间: 2026-04-23 CST
- 适用范围: `/erp-client`、`/erp-server`、数据库迁移、部署联调、首批上线准备
- 当前最高原则: `快速做成可用 ERP`
- 本版本说明:
  - 本文件完全替代旧版“平均铺平台层 + 平均铺治理层”的推进方式
  - 从现在开始，所有 vibecoding 都以“企业真实能用”为唯一主线
  - 功能是否优雅、架构是否漂亮、概念是否先进，都必须让位于 `可用、稳定、可交付`

---

## 1. 文档目的

这不是概念文档，也不是长期愿景展示文档。

这是一份以后持续开发时必须遵守的执行总纲，目的只有五个：

1. 尽快做出一个企业每天能用的 ERP
2. 把当前已经开始发散的页面和功能重新收束
3. 只保留最短交付路径上的模块和能力
4. 让团队知道什么该先做、什么暂时不能做
5. 让后续 vibecoding 不再因为目标过多而变慢

如果后续讨论和本文件冲突，以本文件为准。

## 2. 新的总判断

### 2.1 过去阶段的问题

过去一段时间，项目已经做出了很多资产：

1. Electron 客户端壳层
2. Java Spring Boot + MyBatis-Plus 后端
3. 通用模块页与 `EntityTableView`
4. `ModuleWorkbench`
5. Timeline / Note / Attachment
6. `ext_data`
7. `sys_script`
8. cutover config / reminder / role desk / finance batch / close task 等治理对象

这些不是没价值。

但问题也已经很明显：

1. 页面开始变乱
2. 首页、设置页、模块页都在承载太多事情
3. 治理层长得比业务主线还快
4. 用户“每天怎么完成一单业务”这条主路径还不够顺
5. 一个企业级工具真正要用的那些能力，还没有系统性收口

### 2.2 新的结论

从现在开始，项目不再追求“功能面越来越大”，而是追求：

`先成为一个可以日常真实使用的 ERP，再慢慢成为一个很强的 ERP`

换句话说：

1. 先可用
2. 再稳定
3. 再补深度
4. 最后才补高级平台化

## 3. 当前工程的真实状态

### 3.1 当前已经有的核心资产

基于当前仓库代码，已经形成的可复用资产如下：

#### 客户端层

1. Electron 主客户端已经可作为唯一桌面入口
2. 全局 `DashboardView`、`SettingsView`、`CommandPalette`
3. `ModuleWorkbench` 已经成为主模块容器
4. `EntityTableView` 已经承担大量通用 CRUD 能力
5. 多个核心模块已经存在专用工作台：
   - `ResPartnersView`

   - `PurchaseOrdersView`
   - `StockPickingsView`
   - `AccountInvoicesView`

7. 快捷键、设置页、桌面通知链已经存在基础能力

#### 后端层

1. `ext_data` 已经落到核心表
   - `res_partner`
   - `sale_order`
   - `purchase_order`
   - `account_move`
2. `sys_script` 和最小服务端脚本执行器已经存在
3. `TimelineController` / `TimelineService` 已存在
4. `ReminderService` 已存在并且已接业务提醒
5. `CutoverConfigService`、`CutoverOpsRecordService` 已存在
6. cutover 治理对象已落表：
   - `sys_cutover_config`
   - `sys_cutover_role_task`
   - `sys_cutover_finance_review`
   - `sys_cutover_finance_result_pack`
   - `sys_cutover_close_task`

#### 数据库层

1. 迁移脚本体系已经成型
2. 平台层首批迁移已落
3. cutover 治理对象迁移已落
4. Timeline / Note / Attachment 索引基础已具备

### 3.2 当前最强的地方

当前工程最强的地方，不是业务闭环，而是“基础骨架”：

1. 模块框架已经足够大
2. 前后端生成式模块结构已经成型
3. Electron 桌面壳层已经统一
4. 平台能力不是空白，而是已经有第一层

### 3.3 当前最差、最拖慢速度的地方

当前最欠缺的，不是再缺几个治理对象，而是下面这些：

1. 页面信息架构不够克制
2. 详情页标准不统一
3. 一个对象没有唯一主页面
4. 主业务链还没有被压成“日常能顺畅操作”的体验
5. 打印、导入、导出、权限、部署、错误恢复这些企业级刚需还不够成体系

### 3.4 对当前工程的最终判断

当前项目更像：

`已经拥有不少中后台和平台骨架，但还没有收敛成一个真正顺手的企业 ERP`

所以接下来主线必须改变：

1. 少谈“还要不要再加一个平台层概念”
2. 多谈“一个企业今天能不能用这个系统走完订单、采购、开票、付款、追溯”

## 4. 从现在开始的唯一主线

以后所有 vibecoding 都围绕这句话推进：

`先把 NEKO_ERP 做成一个企业真正能用的桌面 ERP。`

这句话拆开就是：

1. 核心主数据能维护
2. 销售链能跑
3. 采购链能跑
4. 发票/付款/关账能跑最小闭环
5. 每个核心对象有清晰详情页
6. 关键文档能挂上去
7. 关键操作不容易把用户弄乱
8. 构建、部署、验证都可重复

以后所有不直接服务以上目标的内容，优先级自动下降。

## 5. 可用 ERP 的定义

### 5.1 什么叫“可用”

所谓“可用”，不是全模块齐全，而是：

1. 企业用户可以每天使用
2. 能稳定保存关键业务数据
3. 能挂附件、合同、发票、凭证
4. 能查回历史上下文
5. 能完成至少一条销售链和一条采购链
6. 遇到问题有最小回退和补录办法

### 5.2 第一批必须做到可用的范围

以后只围绕这些模块做收口：

#### 主数据

1. `resPartner`
2. `resCompany`
3. `productTemplate`
4. `productProduct`
5. `productCategory`
6. `productPricelist`

#### 销售链

1. `saleOrder`
2. `stockPicking`
3. `accountInvoice`

#### 采购链

1. `purchaseOrder`
2. `accountInvoice`
3. `accountPayment`

#### 财务收口

1. `accountMove`
2. `accountPayment`
3. `accountInvoice`

### 5.3 第一批不要求完整但必须有最小能力的部分

1. Timeline
2. Attachment / Contract / Invoice / Supporting Docs
3. 搜索与最近对象重开
4. 快捷键
5. 基础提醒
6. 基础导入导出
7. 基础打印或至少可导出单据文本

## 6. 现在明确不再追的方向

为了速度，从现在开始明确降级以下事项：

1. 不再继续平均铺所有模块
2. 不再把 Dashboard 做成“大一统全景指挥舱”
3. 不再把 Settings 做成“大而全操作台”
4. 不再优先继续扩新的治理对象概念
5. 不再优先做过深的 ERPNext 元数据平台化
6. 不再优先做过深的 Monica 关系网络实验
7. 不再优先做过深的制造、HR、CRM 二层语义
8. 不再优先做“大而全报表引擎”
9. 不再优先追求和 Odoo 全量等价

这些不是永久不做，而是当前不允许抢占主线。

## 7. 新的产品结构规则

页面乱的根源，不是组件太少，而是层级不清楚。

以后产品结构固定为四层，不允许随意扩：

### 7.1 首页 `Dashboard`

首页只做四件事：

1. 当前试点/系统状态总览
2. 今日待处理事项
3. 高风险入口
4. 快速跳转

首页不负责承载：

1. 大量治理明细
2. 大量配置项
3. 大量二级统计
4. 大量批处理细节
5. 默认展开所有治理范围

首页治理面板新增硬规则：

1. 默认只展示最高优先级的 `3-5` 个范围
2. 次级范围必须折叠在同一面板内，而不是继续摊成更多卡片
3. 当前视图导出必须和当前可见条目一致，不能导出“用户没看到的整批数据”
4. 面板如果没有高优先数据，应该直接显示“已全部展开”或“当前无异常”，不要靠大量空卡片占位
5. 首页头部动作必须拆成“主动作常驻 + 次级打包动作收起”，不能再出现动作墙
6. `Dashboard` 默认只保留风险和审核相关的核心治理面板，批处理、责任台、财务批量台必须通过摘要卡再展开

### 7.2 模块页 `ModuleWorkbench`

模块页只做四件事：

1. 列表
2. 筛选
3. 快速新建
4. 进入详情

模块页可带最小治理摘要，但不能压过主业务操作。

### 7.3 详情页 `Record Detail`

以后一个业务对象必须有唯一主详情页。

详情页固定结构：

1. 顶部: 对象标题、状态、主动作
2. 主区: 表单字段
3. 侧区或次级 Tab:
   - Timeline
   - 附件
   - 合同
   - 发票
   - 操作记录
   - 关联对象
4. 底部或次级区:
   - 审计/治理信息

关键规则：

1. 点击表格记录，默认打开详情页
2. 不再在原列表里无限展开
3. 不允许一个对象在三个不同面板里分别承担主详情职责
4. 只要前端已经知道目标记录 `id`，跳转就必须使用 `detailId`
5. `focusField/focusValue` 只保留给列表筛选和未知精确记录的检索场景
6. 详情页默认只展示高优先级动作、风险和证据入口，次级导出/交接/治理动作统一收敛到专门区域
7. 主链对象默认不重复摊开多个“关联对象卡片墙”，文档走 `Document Hub`，关系和来源走 `Traceability Center`
8. 详情页优化优先级永远高于新增首页卡片和新增治理面板
9. 跨模块跳转不能裸跳模块首页，必须默认落到目标模块最相关的工作区 section
10. 只要跳转来自另一业务模块，就必须尽量保留 `relatedTo`，让用户知道自己是从哪条链路过来的
11. 只要跳转处于首批切换链，就必须尽量保留 `chainKey`，避免销售链、采购链和主数据链上下文丢失
12. `Dashboard / Settings / ActivityCenter / ReminderSummary / PilotReviewQueue / CommandPalette` 这些总览入口也必须复用同一套导航规则，不能各写各的跳转语义

### 7.4 设置页 `Settings`

设置页只做：

1. 连接
2. 客户端配置
3. 快捷键
4. 模块开关
5. 回退与切换配置
6. 少量系统级治理

设置页不是业务操作台。

设置页治理面板新增硬规则：

1. 默认只展示最值得处理的责任桶、风险范围、批处理范围
2. 桶内条目默认只展示最急的少数项
3. 必须允许展开全部，但展开入口仍留在同一面板内
4. 不允许把设置页做成“第二个 Dashboard”
5. cutover 大区里的重型治理面板必须先经过摘要卡，再决定是否展开完整台面
6. `Rollback Closed Loop` 这类长链路明细默认收起，只保留摘要和入口

## 8. 新的技术策略

### 8.1 后端策略

后端现在的策略不是“大重构”，而是“保守复用”：

1. 保持 Java Spring Boot + MyBatis-Plus
2. 保持 PostgreSQL
3. 保持当前迁移机制
4. 保持现有模块实体和基础 CRUD
5. 在已有基础上补“企业可用所需的最短链路”

### 8.2 前端策略

前端现在的策略也不是“重写一整套系统”，而是：

1. 继续复用 `ModuleWorkbench`
2. 继续复用 `EntityTableView`
3. 继续复用已有模块自定义页
4. 把“详情页标准化”放在最高优先级
5. 把首页和设置页降噪
6. 把“默认少而准”的详情页收敛作为长期硬规则，而不是继续在详情页堆新卡片
7. 所有总览型治理面板默认采用 `attention-first compact mode`
8. “展开全部”只能作为补充入口，不能成为默认打开状态

### 8.3 平台层策略

平台层现在只保留最小必要集：

#### 继续保留并推进

1. `ext_data`
2. `timeline`
3. `attachment`
4. `sys_script`
5. `reminder`

#### 但要求改变

这些能力以后只能服务“快速可用 ERP”，不能反过来主导开发节奏。

例如：

1. `ext_data` 是为了少改代码，不是为了搭完整低代码平台
2. `sys_script` 是为了最小 guardrail，不是为了先做脚本生态
3. `timeline` 是为了对象可追溯，不是为了先做关系系统全家桶

## 9. 第一优先级业务闭环

### 9.1 销售链最小闭环

必须尽快做到：

1. 建客户
2. 建产品
3. 建销售订单
4. 挂合同/报价/附件
5. 查看时间线
6. 产生发票
7. 回查来源和状态

### 9.2 采购链最小闭环

必须尽快做到：

1. 建供应商
2. 建采购订单
3. 挂供应商报价/合同/账单附件
4. 产生应付发票
5. 记录付款
6. 回查来源和状态

### 9.3 财务最小闭环

必须尽快做到：

1. 发票可看
2. 付款可看
3. 分录或凭证可看
4. 基础关账任务可跟
5. 能查附件和来源对象

### 9.4 主数据最小闭环

必须尽快做到：

1. 客户/供应商资料可维护
2. 产品资料可维护
3. 公司资料可维护
4. 价格相关主数据可维护
5. 每条主数据有附件与背景备注

## 11. 从成熟 ERP 吸收什么

### 11.1 向 ERPNext 学

只吸收现在真的有用的部分：

1. `ext_data` 混合模型
2. `sys_script` 最小脚本埋点
3. 刚性状态机
4. 统一命名规则

### 11.2 向 Monica 学

只吸收现在真的有用的部分：

1. Timeline
2. 附件和背景知识
3. 最小提醒

### 11.3 向 Odoo 学

继续保留它最现实的长处：

1. 业务对象模型
2. 业务链条分层
3. 模块边界
4. 企业业务语义

### 11.4 向成熟商业 ERP 学

现在最该学的是：

1. 页面层级克制
2. 操作流稳定
3. 打印/导出/审计这些基础能力
4. 不让用户迷路

## 12. 新的开发原则

### 12.1 唯一总原则

`先把企业每天能用的 ERP 做出来。`

### 12.2 细化原则

1. 可用优先于完整
2. 稳定优先于炫技
3. 主链优先于平台想象
4. 详情页优先于首页大屏
5. 真实企业操作优先于概念治理

### 12.3 以后写代码时的判断顺序

以后每个任务都先问：

1. 这件事能不能直接提升企业今天用系统完成业务的成功率？
2. 这件事能不能让页面更清楚、更少迷路？
3. 这件事能不能减少人工补录和沟通成本？

如果三个问题都答不上来，通常就不是当前主线。

## 13. 新的批次推进方式

从现在开始，不再按“平台层 + 治理层 + 切换层平均推进”。

以后按下面顺序推进：

### 批次 A: 页面收敛

目标：

1. 把页面从乱收回来
2. 把详情页结构统一
3. 把首页和设置页瘦身

必须做：

1. 列表点击打开详情页
2. 详情页固定结构
3. Dashboard 降噪
4. Settings 降噪
5. 附件 / 合同 / 发票入口统一
6. Role Desk / Finance Batch / Pilot Batch / Risk Stats 默认改成注意力优先展示
7. 当前视图导出和当前可见数据保持一致
8. Dashboard 头部动作改成主次两层
9. Settings 的 cutover 治理台改成摘要卡 + 按需展开

### 批次 B: 主链可操作

目标：

1. 让销售、采购、发票、付款能顺畅走完

必须做：

1. `resPartner`
2. `product*`
3. `saleOrder`
4. `purchaseOrder`
5. `accountInvoice`
6. `accountPayment`
7. `stockPicking`

### 批次 C: 企业刚需补齐

目标：

1. 把企业真正缺不得的能力补上

必须做：

1. 导入
2. 导出
3. 打印
4. 基础权限
5. 基础部署与运维说明
6. 回退与补录说明

### 批次 D: 稳定化

目标：

1. 构建稳定
2. smoke 稳定
3. 关键主链不炸

### 批次 E: 再回头补平台深度

只有当 A-D 过关，才允许继续加深：

1. 更深 `ext_data`
2. 更深 `sys_script`
3. Query report

## 17. 切换与上线门槛的新定义

以后判断一个阶段是否能进入企业试用，不再只看“做了多少功能”，而是看下面这些是否成立：

1. 主链对象能创建、修改、查询
2. 附件、合同、发票能挂上去
3. Timeline 能看
4. 页面不容易让用户迷路
5. 常见错误不会直接让用户无路可走
6. 构建通过
7. smoke 通过
8. 有最小用户手册
9. 有最小回退方案

缺一项都不能称为“可用 ERP”。

## 21. 最终结论

从现在开始，NEKO_ERP 的主线不再是：

1. 继续平均扩模块
2. 继续平均铺平台层
3. 继续在 Dashboard / Settings 上堆治理面板

而是：

1. 先把页面收敛
2. 先把详情页统一
3. 先把销售、采购、发票、付款做成日常能用
4. 先把附件、时间线、导入导出这些企业刚需补上
5. 再去深化平台层和治理层

唯一目标就是：

`快速做成一个企业真正能用的 ERP。`

以后所有 vibecoding，都必须服从这个目标。
